six-gate verification pipeline — run per Dafny spec
C1Spec + capture
Real verifier run against the committed spec — verbatim stdout+stderr, exit code, ISO-8601 manifest. Nothing summarized from memory. Refuses multi-summary-line output, non-0/4 exit codes, and "out of resource"/"out of memory" markers before a result is ever trusted. dafny_adapter.py
C3Parser hardening
Two vectors against a "clean pass" proving nothing: vacuous preconditions, and one-way weak postconditions. Vector 1: Z3 solver checks the requires conjunction is satisfiable — unsat means every ensures held vacuously. Vector 2: flags ==> ensures clauses lacking a <==> — heuristic, not a proof. dafny_spec_lint.py
C4Spec-Testing Proofs
IronSpec methodology: does the postcondition pin one exact result, or just bound it? Each lemma restates the real ensures as a requires on a hypothetical value, then proves the correct value forced (ACCEPT) or a wrong candidate excluded (REJECT, ensures false) — tests the spec, never evaluates the body. *_stp_suite.dfy
C5Mutation testing
MutDafny-style: perturb one operator or literal, re-verify for real — is that boundary actually load-bearing? Five operator classes: ROR (relational), LOR (logical), AOR (arithmetic), LVR (literal value), COI (negate ensures). Static filters catch trivially-implied mutants before a real Dafny call. dafny_mutate.py
C6NL-dialogue confirmation
Catches a spec/intent mismatch at authoring time, not review time. Plain-English requires/ensures summary, generated mechanically from the signature — a human reads it and signs off it says what was meant. Not self-gradable by design. dafny_nl_summary.py
C2 PROVEN's exclusivity — a structural rule inside the matrix builder itself, not a sequential step: generation fails outright if any record claims strength PROVEN without a real, completed Dafny run behind it.
