PayloadGuard Evidence · System Blueprint · condensed

How a real tool run becomes a claim nobody can round up.

The whole system is one guarded pipeline: authored requirements and primary sources meet real verifier output, and the engine binds them at a strength it will not let you overstate. This is the map — read it before the 1,100-line reference.

Data flow

Four stages, left to right

Nothing is hand-typed downstream of a tool: a capture is the verbatim result of a real run, and the matrix is generated from captures — never edited.

INVARIANT

The core rule the whole architecture defends: a PROVEN label can never appear unless a real, completed Dafny proof produced it — gated on Z3 precondition satisfiability and the false-zero guard before any record is built (ruling R3).

The pipeline

Six gates, C1 → C6

Each gate is a distinct check with its own committed evidence. Five are mechanical; the last is deliberately human and can't be self-graded.

C1

Spec + capture

Run the spec, capture verbatim output and a manifest.

run_verify_*.py · dafny_adapter.py
C2

Evidence binding

Parse output, guard against false-zeros & timeout masking; bind to requirements.

dafny_adapter.py · reconcile.py
C3

Spec lint

Z3 precondition satisfiability (no vacuous proofs) + weak-postcondition heuristic.

dafny_spec_lint.py
C4

Spec-testing proofs

IronSpec-style ACCEPT/REJECT lemmas prove each postcondition is tight, not just held.

*_stp_suite.dfy
C5

Mutation testing

ROR / LOR / AOR / LVR / COI mutants, real-verified; a survivor is a finding.

dafny_mutate.py
C6

NL sign-off human

Plain-English summary of the contract, confirmed by a person against intent.

dafny_nl_summary.py
