# Gate C6 sign-off: `drug_interaction_checker.dfy::CheckInteraction`

## Purpose

Gate C6 (NL-dialogue confirmation) exists to catch a spec/requirement-text
mismatch at authoring time rather than at review time — the same defect
class as Gate 1's original finding. The mechanical step
(`evidence.dafny_nl_summary.summarize_method`) generates a plain-English
reading of a Dafny method's contract; this document is the actual
deliverable the gate calls for: the recorded human decision that the
summary matches intent.

## A real tooling gap found and fixed before this summary could be generated

`CheckInteraction`'s one `requires` clause is written across three
physical lines:

```dafny
requires !(doac == Apixaban &&
           (agent == Rifampicin || agent == Carbamazepine ||
            agent == Phenytoin || agent == Phenobarbital))
```

— the first genuinely multi-line clause this repo has pointed Gate C6's
summary generator at. Every clause in `dosage.dfy` was one line;
`renal_adjustment.dfy` hit an equivalent gap for two `ensures` clauses in
`AssessRenalFunctionFromInputs` (see its own `nl_confirmation_*.md`), and
that time the call was to **reformat** those two clauses to a single
line rather than extend the tool, reasoning that the line break was "a
formatting choice that was mine, not a genuine spec need."

**This time the call went the other way, for a concrete reason.** By
2026-07-10 this exact spec already had a committed Gate C1 capture, a
Gate C4 STP suite (22 lemmas), and a Gate C5 mutation report (962
mutants) all captured against the file *as currently formatted* —
reformatting the `requires` clause, even purely cosmetically, would mean
re-running and re-committing every one of those captures to keep them
honestly bound to the source they claim to verify, for a change with no
semantic content. `evidence/dafny_nl_summary.py` was extended instead:
`_extract_annotated_clauses` now accumulates a clause across multiple
physical lines (ending accumulation at a blank line, a standalone
`//`-comment line, or the next clause keyword — so a free-floating block
comment between two clauses, common in this spec, is never
misattributed as either clause's citation). The existing safety net is
unchanged in spirit: `summarize_method` still cross-checks its own
extraction against `evidence.dafny_spec_lint`'s canonical multi-line-
capable extractor and refuses (`SystemExit`, Tier 1) on any mismatch — a
comment sitting on its own line *inside* a multi-line boolean expression
(rather than between two clauses) still correctly refuses rather than
guess, since that shape is genuinely ambiguous.

Three new tests in `tests/test_dafny_nl_summary.py`:
`test_multiline_clause_is_summarized_correctly_not_refused` (a synthetic
regression, replacing the old blanket-refusal test that this extension
made obsolete), `test_standalone_comment_inside_a_multiline_clause_still_refuses`
(confirms the safety net still catches the genuinely ambiguous case),
and `test_real_ddi_spec_multiline_requires_clause_summarizes_correctly`
(end-to-end, against this real committed spec). 190 tests pass (up from
188).

**Citation note:** none of `CheckInteraction`'s 60 `ensures` clauses
carry an inline `// REQ-DDI-*` citation — unlike `dosage.dfy`'s and
`renal_adjustment.dfy`'s per-clause citation style, this spec's
correctness is validated against `sources/sps-doac-interactions-2024.md`
as a whole lookup table (Gate 1a/1c), not REQ-ID by REQ-ID. Every
`*(no requirement cited)*` below is therefore accurate, not a citation
gap — worth stating explicitly since a reader used to the other two
worked examples' style might otherwise mistake it for one.

## Things worth Steven's specific attention at sign-off

Unlike `dosage.dfy` (3 postconditions) and `renal_adjustment.dfy` (up to
5 per function), this summary is 60 postconditions long — the honest
reflection of a flat lookup-table spec (Gate C4's own finding: this
function's shape doesn't compress into a clean range partition the way
`GStage`'s six boundary clauses do). A few things worth checking
deliberately rather than skimming past:

1. **The precondition's exact scope.** It excludes exactly
   `(Apixaban, {Rifampicin, Carbamazepine, Phenytoin, Phenobarbital})` —
   confirm this still matches the intended Gate 1c Finding 2 resolution
   (apixaban's real source gap for these four enzyme-inducer interactions,
   not assumed safe by omission).
2. **Postconditions 27/28** are the one caller-supplied-flag-conditional
   pair (`hasOtherBleedingRiskFactors` true vs. false for
   Dabigatran+SSRIOrSNRI) — confirm the direction (flag true → dose
   reduction advised, flag false → plain caution) still matches
   `sources/sps-doac-interactions-2024.md`'s intent, not just what the
   STP suite already proves internally consistent.
3. **Postconditions 36 and 40** (`Contraindicated`) are the two
   highest-stakes rows in the whole table — Gate C4's REJECT lemmas
   already prove a weaker `Caution` is excluded for both, but that's a
   proof about internal consistency, not about whether `Contraindicated`
   is still the clinically correct call.
4. Every postcondition reads `*(no requirement cited)*` — confirmed
   above as accurate, not a gap, but worth Steven explicitly agreeing
   that whole-table validation (rather than per-clause REQ-ID citation)
   is the right traceability model for this kind of spec, versus
   `dosage.dfy`/`renal_adjustment.dfy`'s per-clause style.
